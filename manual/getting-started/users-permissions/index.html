Users & permissions

Invite your team, assign employee types and control who can do what.

Most shops have more than one person: sales, buying, floor operators, a bookkeeper. This page shows you how to invite them and control what each one can do.

You need to be an admin to invite people. If you set up the company, you already are. At Acme that's Chase Foster (chase@carbon.ms).

Where this lives in Carbon

People and access live in the Users module, inside Settings. Go to Settings → Users in the left navigation. You'll see everyone in your company and the type assigned to each.

Two simple ideas

Access comes from two things. Keep them separate and it stays easy.

  • A user is one login — one person, one email.
  • An employee type is a role: a bundle of permissions you attach to a user. Set it up once, hand it to everyone who does that job.

Permissions are granted per module (Sales, Purchasing, Items, and so on), with four actions each: View, Create, Update, Delete.

The flow: invite the user, give them an employee type, and that type's permissions decide what they can touch.

Invite your team

You add people by email. They click the link, set a password, and they're in.

Click Invite. Top-right of Settings → Users.

Enter their email and pick a type. Add one or more addresses (separate with commas to invite several at once). Choose the employee type they should get — you can change it later.

They accept. Carbon emails the invite. When they click it and set a password, they're in. Until then their status shows Invited instead of Active.

When someone leaves, deactivate them instead of deleting. That frees the seat but keeps their name on records they created — like Chase Foster staying attached to SO000069 forever.

Choose an employee type

An employee type is the role you hand someone. It carries a default set of permissions, so you don't set up each person from scratch. Here are types a shop like Acme would use.

These names are examples — you create and name your own. The one type that always exists is Administrator: it reaches every module and Settings. Always keep at least one.

Set module permissions

Inside a type, you flip four switches per module: View, Create, Update, Delete.

Open the type. From Settings → Employee Types, click a type (say Sales) and open its Permissions tab. You'll see one row per module.

Tick what the role needs. For a salesperson: full Sales, View on Items (to pick parts onto a quote), and nothing else.

Save. The change applies to everyone with that type right away. No need to edit each person.

Roles side by side

Here's how Acme's common types map across the modules — full access, read-only (View), or none.

Give the least access that does the job. Don't make everyone an Administrator. Start narrow, add a permission when someone hits a wall. Hand out Delete rarely — Carbon prefers to cancel or deactivate records, not erase the history other records depend on.

AI and API access run through this same system. An AI agent or API key inherits the access of the user it acts as, so scope it tight. More in Developing with Carbon.
