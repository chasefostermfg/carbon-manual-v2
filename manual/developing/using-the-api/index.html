Using the API

Carbon is API-first — create a key and make your first call.

Anything you can do by clicking in Carbon, you can also do with code. Carbon is API-first: the web app and the API talk to the same backend. If it's in the app, it's in the API.

In three steps, you'll create a key, install a client, and read real data back out of Carbon.

Where this lives in Carbon

Module: Settings. Create and manage keys under Settings → API Keys. A key is a secret that lets your code log in as you.

If it's in the app, it's in the API. Carbon's UI is built on top of the public API, so every feature is callable from your code the day it ships. That's the one thing to remember.

Loading diagram...

Your code and the web app point at the same backend. The API isn't a copy of your data — it is your data.

Step 1 — Create an API key

Open Settings → API Keys

Go to Settings → API Keys. This page lists every key, who made it, and when it was last used.

Click New API Key and name it

Click New API Key and give it a clear name (like Nightly export). Naming by purpose makes it easy to revoke the right one later.

A key has the same permissions as the user who made it. If you can't approve purchase orders in the app, your key can't either. See Users & permissions.

Copy the secret right away

Carbon shows the full key value once. Copy it now and store it somewhere safe — a secrets manager or password manager. Lose it and you just delete the key and make a new one.

Treat a key like a password. Anyone with it can act as you. Never put it in client-side code, a public repo, or a screenshot. Keep it in an environment variable, and delete any key the moment you think it leaked.

API key or OAuth?

Two ways to prove who's calling. Pick by whose data you're touching.

An API key authenticates you (or one service account). Best for the work most readers do:

  • A nightly script that exports your orders.
  • A backend that pushes shipments into Carbon.
  • One-off migration or reporting jobs.

Send it as a header. Carbon reads your company from the key automatically:

OAuth is for apps that act on behalf of other Carbon users — a product where its users connect their Carbon account. Your app sends each user through a consent screen and gets a short-lived token scoped to that user.

Automating your own company? Use an API key and skip OAuth. Building an app other companies install? The full OAuth flow is in the Developer Docs.

Rule of thumb: automating your data → API key. Building an app other customers log into → OAuth.

Step 2 — Install a client library

You can call Carbon with raw HTTP, but the official client libraries handle auth, paging, and errors for you. Reach for these first.

LanguageInstallGood for
TypeScript / JavaScriptnpm i @carbon/apiWeb apps, Node, edge functions
Pythonpip install carbon-apiData work, scripts, notebooks, ETL
.NETdotnet add package Carbon.ApiC# services and tools

Always-current package names and install commands live in the Clients section of the Developer Docs.

Step 3 — Make your first call

To prove your key works, read something — don't write. You'll list items, then fetch one sales order: the Counterweight Hook (CWH-A36-15) and its order, SO000069.

Read before you write. Read endpoints can't change anything, so they're safe to learn on. When you're ready to write (creating a quote line, releasing a job), use a test or demo company first — never your live data.
