Anything you can do by clicking in Carbon, you can also do with code. Carbon is **API-first**:
the web app and the API talk to the *same* backend. If it's in the app, it's in the API.

In three steps, you'll create a key, install a client, and read real data back out of Carbon.

## Where this lives in Carbon

**Module: Settings.** Create and manage keys under **Settings → API Keys**. A key is a secret
that lets your code log in as you.

<Callout type="info">
  **If it's in the app, it's in the API.** Carbon's UI is built *on top of* the public API, so
  every feature is callable from your code the day it ships. That's the one thing to remember.
</Callout>

  chart={`flowchart LR

Your code and the web app point at the **same** backend. The API isn't a copy of your data — it
*is* your data.

<Step>
### Click New API Key and name it

Click **New API Key** and give it a clear name (like `Nightly export`). Naming by purpose makes
it easy to revoke the right one later.

A key has the **same permissions as the user who made it**. If you can't approve purchase orders
in the app, your key can't either. See [Users & permissions](/manual/getting-started/users-permissions).
</Step>

<Step>
### Copy the secret right away

Carbon shows the full key value **once**. Copy it now and store it somewhere safe — a secrets
manager or password manager. Lose it and you just delete the key and make a new one.

<Callout type="warn">
  **Treat a key like a password.** Anyone with it can act as you. Never put it in client-side
  code, a public repo, or a screenshot. Keep it in an environment variable, and delete any key
  the moment you think it leaked.
</Callout>
</Step>

</Steps>

## API key or OAuth?

Two ways to prove who's calling. Pick by **whose** data you're touching.

<Tabs items={["API key", "OAuth"]}>

<Tab value="API key">
An **API key** authenticates *you* (or one service account). Best for the work most readers do:

- A nightly script that exports your orders.
- A backend that pushes shipments into Carbon.
- One-off migration or reporting jobs.

Send it as a header. Carbon reads your company from the key automatically:

```http
Authorization: Bearer carbon_sk_your_key_here
```
</Tab>

<Tab value="OAuth">
**OAuth** is for apps that act *on behalf of other Carbon users* — a product where *its* users
connect *their* Carbon account. Your app sends each user through a consent screen and gets a
short-lived token scoped to that user.

Automating your own company? Use an **API key** and skip OAuth. Building an app other companies
install? The full OAuth flow is in the Developer Docs.
</Tab>

</Tabs>

<Callout type="info">
  **Rule of thumb:** automating *your* data → **API key**. Building an app *other* customers log
  into → **OAuth**.
</Callout>

## Step 3 — Make your first call

To prove your key works, **read** something — don't write. You'll list items, then fetch one
sales order: the Counterweight Hook (`CWH-A36-15`) and its order, **SO000069**.

<Steps>

<Step>
### Put your key in the environment

Don't hard-code the secret. Set it once so your code reads it at runtime:

```bash
export CARBON_API_KEY="carbon_sk_your_key_here"
```
</Step>

<Step>
### List your items

Ask Carbon for a page of items. Rows back means your key works.

<Tabs items={["TypeScript", "Python"]}>

<Tab value="TypeScript">
```ts
import { CarbonClient } from "@carbon/api";

const carbon = new CarbonClient({
  apiKey: process.env.CARBON_API_KEY,
});

// Read the first 10 items in your company.
const { data: items } = await carbon.items.list({ limit: 10 });

for (const item of items) {
  console.log(item.readableId, "—", item.name, `(${item.itemType})`);
}
// CWH-A36-15 — Counterweight Hook (Part)
// A36-STEEL-PLT-1".96x240 — A36 Steel Plate 1" (Material)
```
</Tab>

<Tab value="Python">
```python
import os
from carbon_api import CarbonClient

carbon = CarbonClient(api_key=os.environ["CARBON_API_KEY"])

# Read the first 10 items in your company.
items = carbon.items.list(limit=10)

for item in items.data:
    print(item.readable_id, "—", item.name, f"({item.item_type})")
# CWH-A36-15 — Counterweight Hook (Part)
# A36-STEEL-PLT-1".96x240 — A36 Steel Plate 1" (Material)
```
</Tab>

</Tabs>

<Step>
### Fetch one sales order

Now grab a single record: **SO000069**, Radiant Aerospace's order for 500 Counterweight Hooks.

<Tabs items={["TypeScript", "Python"]}>

<Tab value="TypeScript">
```ts
// Look up one sales order by its readable number.
const { data: order } = await carbon.salesOrders.get("SO000069");

console.log(order.customerName); // Radiant Aerospace
console.log(order.orderTotal);   // 36820
console.log(order.status);       // Completed

for (const line of order.lines) {
  console.log(line.itemReadableId, line.saleQuantity, "@", line.unitPrice);
  // CWH-A36-15 500 @ 73.14
}
```
</Tab>

<Tab value="Python">
```python
# Look up one sales order by its readable number.
order = carbon.sales_orders.get("SO000069").data

print(order.customer_name)  # Radiant Aerospace
print(order.order_total)    # 36820
print(order.status)         # Completed

for line in order.lines:
    print(line.item_readable_id, line.sale_quantity, "@", line.unit_price)
    # CWH-A36-15 500 @ 73.14
```
</Tab>

<Callout type="tip">
  **Read before you write.** Read endpoints can't change anything, so they're safe to learn on.
  When you're ready to write (creating a quote line, releasing a job), use a **test or demo
  company** first — never your live data.
</Callout>

## Good to know

<Accordions type="single">

<Accordion title="Pagination — getting more than one page">
List endpoints return one page at a time. Pass `limit` (how many) and `offset` (how many to
skip): `limit: 50, offset: 50` gets the second page of fifty. A `count` tells you the total.
</Accordion>

<Accordion title="Readable IDs vs. internal IDs">
Records have two IDs: a friendly **readable ID** (`SO000069`, `CWH-A36-15`) and an internal
**record ID** used in URLs (`so_Fx6FCeSpx9sxut9Qns6y91`). Fetch by either. Readable IDs are for
people; internal IDs are stable references for linking records.
</Accordion>

<Accordion title="Errors and rate limits">
Standard HTTP codes: `401` key missing or wrong, `403` key lacks permission, `404` record not
found, `429` too many requests. The clients surface these as typed errors. On a `429`, wait and
retry.
</Accordion>

<Accordion title="Dates and currency">
Dates are ISO 8601 strings (`2026-06-11`). Money comes back as plain numbers in the base
currency (USD for Acme) — `36820` is **$36,820.00**. Format it for display on your side.
</Accordion>

</Accordions>
