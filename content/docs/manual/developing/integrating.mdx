Most ERP systems are a closed box. Carbon is the opposite: everything in the app runs on the
same public [API](/manual/overview/glossary) you can use too. So you can connect Carbon to your
other tools, or build the feature you wish existed.

## Where this lives in Carbon

This is part of the **Developer** module. You set up keys and connections in
**Settings → Integrations** and **Settings → API**.

There are three ways to connect:

  columns={[

    { key: "when", label: "Use it when" },
  ]}

Most integrations use the first two together: a **webhook** tells you something happened, then an
**API** call gets the details.

<Callout type="tip">
  New here? Read [Using the API](/manual/developing/using-the-api) first to make a key and your
  first call. Then come back to wire things together.
</Callout>

## Webhooks: Carbon tells you when things happen

A **webhook** lets Carbon call *you*. Instead of asking "anything new yet?" over and over, you
give Carbon a URL once. Carbon sends an HTTP request to that URL whenever an **event** you care
about happens. Think of it as Carbon tapping you on the shoulder.

<Mermaid
  chart={`flowchart LR
  E[Event: sales order confirmed] --> CARBON[Carbon]
  CARBON --> POST[HTTP POST to your URL]
  POST --> APP[Your handler]
  APP --> ACT[Notify Slack / portal / shipping]`}
/>

<Steps>
<Step>
**Pick the events you care about.** For a shop like Acme: a quote sent, a sales order confirmed,
a job completed, a shipment shipped, an invoice paid. Subscribe only to what you'll act on.
</Step>
<Step>
**Give Carbon a URL.** Your endpoint is a small web service that accepts an HTTP `POST`. Carbon
sends a short JSON body saying what happened and which record it was.

```json
{
  "event": "salesOrder.confirmed",
  "recordId": "so_Fx6FCeSpx9sxut9Qns6y91",
  "readableId": "SO000069",
  "occurredAt": "2026-06-11T20:27:06.171Z"
}
```

The payload is small on purpose: it tells you *what* and *which record*, not every field. Use the
`recordId` to fetch the full, current details with an API call.
</Step>
<Step>
**Check it's really from Carbon.** Carbon signs each request. Verify the signature before you act.
Then reply fast with a `2xx` status and do the slow work after, so Carbon doesn't think delivery
failed and retry.
</Step>
<Step>
**Act on it.** Fetch the details, then push them where they need to go. For `SO000069`, that could
mean grabbing the line for 500 Counterweight Hooks at `$73.14` and posting "New order: $36,820 from
Radiant Aerospace" to Slack.
</Step>
</Steps>

<Callout type="tip">
  If you're running a script every five minutes to ask "anything new?", use a webhook instead.
  Polling wastes calls and adds delay; webhooks fire the instant the thing happens.
</Callout>

## Build apps on the API

When you need to *do* something to Carbon, you use the API. The same API powers the whole app, so
anything you can do by clicking, you can do with code.

<DataTable
  columns={[
    { key: "want", label: "Build this" },
    { key: "reads", label: "Reads" },
    { key: "writes", label: "Writes" },
  ]}
  rows={[
    { want: "Customer order portal", reads: "Orders, shipments, invoices for one customer", writes: "New quotes from their cart" },
    { want: "Shop-floor TV dashboard", reads: "Open jobs, operation status, work-center load", writes: "—" },
    { want: "Nightly cost report", reads: "Jobs, completed quantities, labor seconds", writes: "—" },
    { want: "Sync orders from your webstore", reads: "Item catalog, customer list", writes: "Customers, sales orders" },
  ]}
/>

A read is as direct as it sounds. Ask for sales order `SO000069`, get its lines back. A webhook
handler can use the `recordId` it received:

```ts
import { CarbonClient } from "@carbon/api";

const carbon = new CarbonClient({ apiKey: process.env.CARBON_API_KEY });

const { data: order } = await carbon.salesOrders.get("SO000069");
```

```json
{
  "data": {
    "salesOrderId": "SO000069",
    "status": "Completed",
    "orderTotal": 36820,
    "lines": [
      { "itemReadableId": "CWH-A36-15", "saleQuantity": 500, "unitPrice": 73.14 }
    ]
  }
}
```

<Callout type="info">
  The field names you see here (`salesOrderId`, `unitPrice`) are the same ones used inside the app.
  There's no separate "integration data model" to learn.
</Callout>

## Example integrations: clone and adapt

Carbon ships real, open-source examples. They're the fastest way to go from "I think this is
possible" to a working pattern.

**quote-configurator** asks the customer a few questions (size, material, finish) and turns the
answers into a priced quote automatically. Acme already has one in its item list: the **Flame Cut
Part Configurator** (`FLAME CONFIG`). The app computes a price and creates a quote through the API,
just like the `Q000061` quote you'd build by hand in [Quoting](/manual/selling/quoting).

**upload-3d-model** attaches a **3D model** to an [item](/manual/overview/glossary) so anyone can
see the part. The Counterweight Hook already carries one: `counterweight hook 3d.stl`. The repo
shows how to upload a model file and link it to the right item.

<Callout type="tip">
  Treat these as templates, not products. Clone one, point it at your own company with your API key,
  and reshape it. Same field names as the app, so the gap is usually small.
</Callout>

## The QuickBooks sync, end to end

The most common integration is keeping your **accounting** system in step. Acme runs its books in
[QuickBooks](/manual/accounting/overview). Carbon owns what was made, shipped and billed;
QuickBooks owns the general ledger. The sync keeps them agreeing without double data entry, and
it's a textbook example of the webhook-plus-API pattern.

<Mermaid
  chart={`flowchart LR
  INV[Invoice AR000025 posted] --> HOOK[Webhook fires]
  HOOK --> FETCH[Fetch invoice via API]
  FETCH --> MAP[Map customer and amounts]
  MAP --> QB[Create invoice in QuickBooks]`}
/>

<Steps>
<Step>
**Connect once.** In **Settings → Integrations**, authorize Carbon to talk to your QuickBooks
company. The link persists; you don't log in every time.
</Step>
<Step>
**Match existing records.** Both systems have customers and items, so the sync maps them: Carbon's
`Radiant Aerospace` links to the matching QuickBooks customer. Getting this right stops the same
customer appearing twice.
</Step>
<Step>
**Carbon posts an invoice.** When Acme issues `AR000025`, a webhook fires and the sync wakes up.
</Step>
<Step>
**The sync writes it across.** It pulls the full invoice (`$36,570` subtotal, `$250` shipping,
`$36,820` total) and creates the matching invoice in QuickBooks. When Radiant pays, that status can
flow over too.
</Step>
</Steps>

<Callout type="warn">
  Decide who owns each piece of data before you connect anything. At Acme, Carbon is the source of
  truth for invoices and QuickBooks just receives them. Two systems editing the same field is the
  fastest way to a reconciliation headache. Pick one owner per field and let the sync flow one way.
</Callout>
