Carbon has a built-in **MCP server**. It lets an AI assistant — Claude, ChatGPT, or your
code editor — read and act inside your Carbon company using plain English. You ask a
question; the assistant does the clicking for you.

## Where this lives in Carbon

This is part of the **Developer** module — the same place as the [API](/manual/developing/using-the-api).
Carbon hosts the server for you at **app.carbon.ms/mcp**. Nothing to install. You connect an
assistant to that URL, and it can run Carbon on your behalf.

<Callout type="info">
  The API is the raw machinery — every Carbon action as a callable endpoint. **MCP** wraps
  that machinery so an AI assistant can drive it. Want to wire Carbon to another system
  instead of talking to it? See [Integrating with other systems](/manual/developing/integrating).
</Callout>

## What MCP is, in one line

**MCP** ([Model Context Protocol](/manual/overview/glossary)) is a USB port for AI: any
assistant that "speaks MCP" can plug into any system that exposes an MCP server — no custom
code in between.

Carbon's server exposes about **1,188 tools** across **15 modules** (Sales, Purchasing,
Items, Production, Inventory, Quality, Accounting, and more). In short: almost anything you
can click in Carbon, an assistant can do through MCP.

## The three meta-tools

A thousand-plus tools is too many to hand an assistant at once. So Carbon hides them behind
just **three** tools. The assistant uses these to find and run everything else on demand.

<DataTable
  columns={[
    { key: "tool", label: "Tool", mono: true },
    { key: "does", label: "What it does" },
  ]}
  rows={[
    { tool: "search_tools", does: "Find the right tools by keyword, module, or safety class." },
    { tool: "describe_tool", does: "Show the inputs and outputs for one tool." },
    { tool: "call_tool", does: "Run a tool and return the result." },
  ]}
/>

The pattern is always: **search → describe (if needed) → call.** You never see these steps —
you just see the answer. Ask *"what's the on-hand quantity of A36 steel?"* and the assistant:

<Steps>
<Step>
**Finds the tool.** Runs `search_tools({ query: "inventory quantity" })` and gets back
candidates like `inventory_getItemQuantities`.
</Step>
<Step>
**Checks the inputs.** If unsure, runs `describe_tool` to read which fields are required.
</Step>
<Step>
**Calls it.** Runs `call_tool` with the A36 plate's item ID, then answers in a sentence.
</Step>
</Steps>

## Three safety classes

Reading a quantity is harmless; deleting a sales order is not. So Carbon tags **every** tool
with one of three safety classes. You can filter by them — for example,
`search_tools({ classification: "READ" })` returns only the look-but-don't-touch tools.

<Callout type="warn">
  **Treat WRITE and DESTRUCTIVE tools like a teammate with edit access.** The assistant is
  changing your live data, just like a colleague clicking the buttons. When exploring, keep
  it to **READ** tools. Turn on WRITE access deliberately, for one task, with a human checking
  the result.
</Callout>

## What you can actually ask

You don't memorize tool names. You describe what you want; the assistant finds the tools.
Here are real asks against Acme's data.

<Tabs items={['Read a value', 'Find a problem', 'Take an action']}>
<Tab value="Read a value">

> **"What's the on-hand quantity of A36 steel?"**

A pure **READ**. Nothing changes. More safe reads:

- "Show every job for **Radiant Aerospace** that isn't finished yet."
- "What did we invoice Radiant on **AR000025**, and is it paid?"
- "List open purchase orders to **Ryerson**."

</Tab>
<Tab value="Find a problem">

> **"Which sales orders have been stuck in Needs Approval for 3+ days?"**

The assistant reads the order list, filters by status and date, and hands you the offenders.
This is MCP's sweet spot — a question that would take several filtered screens, answered in
one sentence. More triage:

- "Any purchase orders in Draft older than a week?" (**PO000056** to Ryerson would surface.)
- "List jobs past their due date that aren't complete."
- "Which invoices are overdue and unpaid?"

</Tab>
<Tab value="Take an action">

> **"Release every sales order stuck in Needs Approval for 3+ days."**

Now the assistant *reads* to find the stuck orders, then issues a **WRITE** call to release
each one. Approve WRITE access first, and ask to see the list before it acts. More actions:

- "Create a quote for **500 Counterweight Hooks** for Radiant Aerospace at **$73.14** each."
- "Bump the priority on job **J000099** to the top of the queue."

</Tab>
</Tabs>

<Callout type="tip">
  Good prompts are specific. "Release the stuck orders" works; "Release sales orders in
  *Needs Approval* with an order date 3+ days ago, **and show me the list first**" gives the
  assistant a clean target and you a checkpoint.
</Callout>

## How the assistant reaches Carbon

On every request the assistant signs in as a real Carbon user, and Carbon enforces that
user's permissions on every call.

<Mermaid
  chart={`flowchart LR
  A[You ask in plain English] --> B[AI assistant]
  B --> C[Carbon MCP server]
  C --> D[Auth and permissions check]
  D --> E[Carbon API and database]
  E --> B`}
/>

- **Sign-in** is either an **OAuth** login (you log in to Carbon once; the assistant gets a
  scoped token) or an **API key** from Carbon's settings — the same key you'd use for the
  [API](/manual/developing/using-the-api).
- **Permissions** come from whoever owns the credentials. If that user can't delete sales
  orders in the app, the assistant can't either. MCP is a new doorway into Carbon, not a way
  around its [permission model](/manual/getting-started/users-permissions).

<Callout type="warn">
  An API key is a password. Store it in your assistant's secret settings (never paste it into
  a chat), give each integration its own key, and revoke a key the moment it's done.
</Callout>

## Connect an assistant

The menu differs by assistant, but the shape is the same: point it at Carbon's MCP URL and
give it credentials.

<Steps>
<Step>
**Make credentials in Carbon.** Open **Settings → API keys** and create a key (or set up
OAuth). Copy it somewhere safe — you won't see it again.
</Step>
<Step>
**Add Carbon as an MCP server.** In your assistant's connectors or MCP section, add a server
pointing at **app.carbon.ms/mcp**. Paste your key, or finish the OAuth sign-in.
</Step>
<Step>
**Confirm it.** The assistant should now list Carbon's three meta-tools —
`search_tools`, `describe_tool`, `call_tool`. That's exactly right; the full catalog stays
hidden behind them.
</Step>
<Step>
**Start with a READ.** Ask something safe, like *"List Radiant Aerospace's open jobs."* If you
get a sensible answer, you're connected. Only then move to WRITE asks, deliberately.
</Step>
</Steps>

Once connected, the assistant exposes Carbon's three meta-tools; everything else stays
hidden behind them and is discovered on demand.

<Callout type="info">
  `call_tool` takes on the safety class of whatever it runs. Calling
  `inventory_getItemQuantities` through it is a READ; calling `sales_deleteSalesOrder` through
  it is DESTRUCTIVE. So the READ-first habit still matters.
</Callout>

## A worked example

Here's the full round-trip behind the on-hand-steel question. You'd only type the first line.

```text
You:  What's the on-hand quantity of A36 steel?

→ search_tools({ query: "inventory item quantity" })
  ← ["inventory_getItemQuantities", "inventory_getItemLedger", ...]

→ describe_tool({ name: "inventory_getItemQuantities" })
  ← { inputs: { itemId, locationId? }, outputs: { quantityOnHand, ... } }

→ call_tool({
    name: "inventory_getItemQuantities",
    arguments: { args: { itemId: "item_Mqv4Jpc9mhgpxTSFfpxjwa" } }
  })
  ← { data: { quantityOnHand: 0, locationId: "loc_CZUGEFrKS1CCvUruMgY9CA" } }

Assistant:  You have 0 LB of A36 Steel Plate 1" on hand at Headquarters.
            Each Counterweight Hook needs 40 LB, so you'd have to buy
            more before you can build any.
```

That last sentence is the real value: the assistant didn't just fetch a number — it tied it
to something you care about. You can't build a hook until you raise another
[purchase order](/manual/buying/purchasing-basics) to Ryerson, like PO000057.

</Callout>
